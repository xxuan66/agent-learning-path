- **避免重复踩坑**：前人踩过的坑，后人不必再踩

## 案例 1：Agent 陷入无限循环

### 问题描述

Agent 在执行搜索任务时，反复调用同一个搜索工具，每次得到相同结果，却不断重复。

### 复现代码

下面的代码复现了无限循环问题：创建一个 Agent 和搜索工具，但没有设置步数限制。Agent 收到"搜索 Python 教程"后，会反复调用搜索工具，每次都得到相同结果，却无法自我终止。

```python
# 有问题的实现
from langchain_openai import ChatOpenAI
from langchain.agents import AgentExecutor, create_tool_calling_agent
from langchain_core.prompts import ChatPromptTemplate

llm = ChatOpenAI(model="gpt-4", temperature=0)
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是一个有帮助的助手"),
    ("human", "{input}"),
])
agent = create_tool_calling_agent(llm, [search_tool], prompt)
agent_executor = AgentExecutor(agent=agent, tools=[search_tool])

result = agent_executor.invoke({"input": "搜索 Python 教程"})

# Agent 的行为：
# Thought: 我需要搜索 Python 教程
# Action: search("Python 教程")
# Observation: 找到10个结果...
# Thought: 我需要搜索 Python 教程
# Action: search("Python 教程")
# ... 无限循环
```

### 原因分析

1. Agent 没有记录"已经搜索过"
2. 提示词没有强调"不要重复"
3. 缺少最大步数限制

### 解决方案

从三个层面解决循环问题：

1. **设置最大步数**：`max_iterations=10`，超过就强制终止
2. **记录动作历史**：检测到重复动作就停止
3. **在提示词中明确禁止**：告诉 Agent 不要重复执行相同操作

```python
# 1. 添加步数限制
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    max_iterations=10  # 最多10步
)

# 2. 添加动作历史
class AgentWithMemory:
    def __init__(self):
        self.history = []
    
    def should_stop(self, action):
        if action in self.history:
            return True
        self.history.append(action)
        return False

# 3. 在提示词中强调
prompt = """
重要：不要重复执行相同的操作。
如果已经执行过某个操作，直接使用之前的结果。
"""
```

### 教训

- **永远设置最大步数**：防止无限循环
- **记录动作历史**：避免重复
- **在提示词中明确禁止**：预防为主

## 案例 2：工具返回格式不一致

### 问题描述

同一个工具在不同情况下返回的格式不同，导致 Agent 解析失败。

### 复现代码

下面的问题是：天气工具在城市存在时返回自然语言（`"{city}今天晴天，温度25度"`），在城市不存在时返回错误格式（`"Error: 城市不存在"`）。Agent 期望的是统一的格式，遇到不同格式就无法正确解析。

```python
@tool
def get_weather(city: str) -> str:
    """获取天气信息"""
    # 有时返回成功格式
    if city in database:
        return f"{city}今天晴天，温度25度"
    # 有时返回错误格式
    else:
        return "Error: 城市不存在"
```

Agent 收到不同格式后无法正确处理。

### 原因分析

1. 工具返回格式不统一
2. Agent 期望固定格式
3. 错误处理不完善

### 解决方案

统一使用 JSON 格式返回，包含明确的 `status` 字段（`success` 或 `error`）。这样 Agent 就可以根据 status 判断操作是否成功，不会被格式搞混。

```python
# 统一返回格式
@tool
def get_weather(city: str) -> str:
    """获取天气信息"""
    import json
    
    if city in database:
        return json.dumps({
            "status": "success",
            "data": {
                "city": city,
                "weather": "晴天",
                "temperature": 25
            }
        })
    else:
        return json.dumps({
            "status": "error",
            "message": f"城市 '{city}' 不存在"
        })
```

### 教训

- **工具返回格式必须统一**：使用 JSON 或固定结构
- **错误和成功分开处理**：明确的 status 字段
- **在工具文档中说明格式**：让 Agent 知道期望什么

## 案例 3：Token 消耗失控

### 问题描述

一个简单的查询任务，Agent 消耗了 10000+ tokens，成本远超预期。

### 复现代码

Token 消耗失控的典型场景：每次调用 Agent 时，系统提示（2000 tokens）+ 工具描述（1500 tokens）+ 历史对话（5000 tokens，不断累积）+ 响应（1500 tokens）= 总共 10000 tokens，其中大部分是重复或不必要的内容。

```python
# Agent 在每次调用时都带上完整历史
result = agent.run("今天天气怎么样")

# 实际的 LLM 调用：
# System Prompt: 2000 tokens
# Tool Descriptions: 1500 tokens (每次都带)
# History: 5000 tokens (累积的历史)
# Response: 1500 tokens
# Total: 10000 tokens
```

### 原因分析

1. 每次调用都携带完整系统提示
2. 工具描述太长
3. 历史对话不断累积
4. Agent 回复过于冗长

### 解决方案

四个方向同时优化：

1. **精简系统提示**：从长篇大论压缩到一两句话
2. **压缩工具描述**：工具的 description 越短越好
3. **截断历史**：只保留最近 N 轮对话，旧的丢弃
4. **限制回复长度**：在提示词中要求"简洁回答"

```python
# 1. 精简系统提示
system_prompt = "你是一个助手。"  # 而不是长篇大论

# 2. 压缩工具描述
tools = [
    Tool(
        name="search",
        description="搜索",  # 简短描述
        func=search_func
    )
]

# 3. 截断历史
def truncate_history(history, max_tokens=2000):
    # 只保留最近的对话
    return history[-max_tokens:]

# 4. 限制回复长度
prompt += "\n请简洁回答，不超过200字。"
```

### 教训

- **监控 Token 使用**：设置告警
- **精简提示词**：越短越好
- **控制历史长度**：滑动窗口
- **限制输出长度**：要求简洁

## 案例 4：Agent 幻觉导致错误操作

### 问题描述

Agent "幻觉"出一个不存在的 API，然后尝试调用，导致错误。

### 复现代码

```python
# Agent 有这些工具
tools = [search, calculator, send_email]

# Agent 的行为：
# Thought: 我需要查询用户的订单
# Action: get_order  # 这个工具不存在！
# Error: Tool 'get_order' not found
```

### 原因分析

1. Agent 基于训练数据"猜测"工具
2. 没有严格检查工具是否可用
3. 某些框架会"发明"工具名称

### 解决方案

```python
# 1. 在提示词中明确可用工具
prompt = f"""
你只能使用以下工具：
{tool_list}

不要使用上述列表之外的任何工具。
"""

# 2. 严格校验工具名称
def validate_tool_call(action, available_tools):
    if action.tool not in available_tools:
        return f"错误：工具 '{action.tool}' 不存在。可用工具：{available_tools}"
    return None

# 3. 使用函数调用而非自由文本
# OpenAI function calling 比 ReAct 更可靠
```

### 教训

- **明确声明可用工具**：在提示词中列出
- **校验工具调用**：执行前检查
- **优先使用 Function Calling**：比自由文本更可靠

## 案例 5：并发请求导致状态混乱

### 问题描述

多个用户同时使用 Agent，状态互相干扰。

### 复现代码

```python
# 全局 Agent 实例
agent = Agent()

# 用户 A 和用户 B 同时请求
# 用户 A: "我的名字是张三"
# 用户 B: "我的名字是李四"
# Agent: "你好，张三"  # 发给了用户 B！
```

### 原因分析

1. Agent 状态是全局共享的
2. 没有会话隔离
3. 并发控制缺失

### 解决方案

```python
# 每个会话独立的 Agent
class AgentManager:
    def __init__(self):
        self.sessions = {}
    
    def get_agent(self, session_id):
        if session_id not in self.sessions:
            self.sessions[session_id] = Agent()
        return self.sessions[session_id]

# 使用示例
manager = AgentManager()
agent_a = manager.get_agent("user_a")
agent_b = manager.get_agent("user_b")
```

### 教训

- **会话隔离**：每个用户独立状态
- **避免全局变量**：使用请求上下文
- **考虑并发安全**：加锁或使用线程本地存储

| 问题 | 发生频率 | 严重程度 | 预防措施 |

## 最佳实践

1. **防御性编程**：假设一切都会出错
2. **充分测试**：边界情况、异常输入
3. **监控告警**：及时发现问题
4. **逐步上线**：先小范围测试
5. **记录日志**：便于问题排查
