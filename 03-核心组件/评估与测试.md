# Agent 评估与测试

> 如何衡量 Agent 的能力？如何确保 Agent 在生产环境中可靠？

---

## 为什么需要评估

想象你招聘了一个新员工（Agent），你怎么判断他是否胜任？你需要：

- **验证能力**：Agent 是否能完成预期任务？——就像试用期考核
- **比较方案**：不同设计、不同模型哪个更好？——就像面试对比多个候选人
- **发现问题**：Agent 哪里容易出错？——就像员工绩效分析，找到薄弱环节
- **持续改进**：有了基线数据，才能量化改进了多少——就像 KPI 对比

**一句话总结**：没有评估，你就不知道 Agent 到底行不行，改进也无从下手。

---

## 评估维度（从哪 5 个角度考察 Agent）

评估一个 Agent，就像评估一个员工，要从多个维度打分，不能只看结果。

### 1. 任务完成度 —— 事情做完了没有？

最直观的维度：Agent 是否成功完成了用户要求的任务？

| 等级 | 描述 | 类比 |
|------|------|------|
| ✅ 完全完成 | 任务目标完全达成 | 员工按时高质量交付 |
| ⚠️ 部分完成 | 完成了主要目标，但有遗漏 | 做完了 80%，漏了细节 |
| ❌ 未完成 | 任务失败或结果错误 | 交了个错误的东西 |

### 2. 效率 —— 做这件事花了多少资源？

同样完成任务，有的 Agent 调用了 3 次大模型就搞定，有的调了 20 次。效率直接影响成本和用户体验。

我们需要追踪以下指标：

| 指标 | 含义 | 为什么重要 |
|------|------|-----------|
| LLM 调用次数 | Agent 一共调用了几次大模型 | 调用越多，响应越慢，成本越高 |
| Token 消耗 | 总共消耗了多少文字量 | 直接决定 API 费用 |
| 工具调用次数 | 调用了多少次外部工具（搜索、计算等） | 过多可能说明 Agent 思路混乱 |
| 总耗时 | 用户从提问到拿到结果用了多久 | 直接影响体验 |
| 成本 | 花了多少钱（美元） | 商业化必须关注 |

### 3. 准确性 —— 给出的答案对不对？

完成任务不等于答案正确。准确性又分为三层：

- **事实准确性**：Agent 说的信息是否真实？比如问"中国的首都是哪里"，回答"上海"就是事实错误
- **逻辑准确性**：推理过程是否合理？比如计算 12×12，过程对了但最后写错结果
- **格式准确性**：输出格式是否符合要求？比如要求返回 JSON，结果返回了纯文本

### 4. 鲁棒性 —— 遇到意外情况会不会崩？

真实使用中，用户输入千奇百怪。鲁棒性就是考察 Agent 在异常情况下的表现：

- **输入格式变化**：用户少打了一个字、用了简写，Agent 还能理解吗？
- **工具返回异常**：搜索工具返回空结果或报错，Agent 能优雅处理吗？
- **网络超时**：某个工具响应慢了，Agent 会卡住还是有降级方案？
- **边界情况**：输入超长、输入为空、输入为特殊字符时的表现

### 5. 安全性 —— 会不会产生有害输出？

这是上线前的最后一道关：

- **系统提示词泄露**：用户能否通过诱导让 Agent 说出自己的"工作指令"？
- **危险操作**：Agent 会不会在没有确认的情况下执行删除数据、发邮件等敏感操作？
- **不当内容**：Agent 是否会产生歧视性、攻击性或虚假信息？

## 三种评估方法

### 1. 人工评估 —— 最准确但最贵

让真人来评价 Agent 的输出质量。就像老师批改作文，最靠谱但最耗时。

**操作流程**：
1. 准备 20-50 个有代表性的测试用例（覆盖常见场景和边界场景）
2. 让 Agent 逐个处理这些用例，收集输出结果
3. 评估者根据统一标准打分（通常 1-5 分）
4. 汇总所有评分，计算平均分和分布情况

**优点**：最准确，能评估主观质量（比如回答是否友好、是否易懂）

**缺点**：成本高，评估者之间标准可能不一致，无法大规模应用

**适用场景**：产品上线前的关键验证、新功能的首次评估

### 2. 自动评估 —— 用 AI 评估 AI

既然人工评估太贵，能不能让另一个 AI 来当评委？这就是自动评估的核心思路。

**工作原理**：你准备好"用户输入"、"Agent 输出"和"评分标准"三个要素，然后让一个强大的大模型（如 GPT-4）根据标准对输出打分。

**工作流程图**：

```
用户输入 "明天北京天气怎么样"
       ↓
被评估的 Agent → 输出 "明天北京晴，气温 15-22°C"
       ↓
评估模型（GPT-4）收到：
  - 用户输入
  - Agent 输出
  - 评分标准（准确性、完整性、格式）
       ↓
评估模型 → 给出 4 分 + 理由 "信息准确，但缺少风力信息"
```

**优点**：成本低，可以跑上千个测试用例，结果可重复

**缺点**：评估模型本身也可能出错，不是 100% 可靠

**适用场景**：大规模回归测试（每次改代码后快速验证有没有破坏旧功能）

### 3. 基准测试（Benchmark）—— 用标准考卷考试

就像高考有统一试卷一样，AI 领域也有标准化的"考试"。这些基准测试提供了一套统一的数据集和评分方式，让不同 Agent 可以公平对比。

**业界常用的基准测试**：

| 基准 | 考什么 | 规模 | 通俗解释 |
|------|--------|------|---------|
| **AgentBench** | 综合能力 | 8 种环境 | Agent 的"高考"，在不同场景下考操作、推理、规划 |
| **WebArena** | 网页操作 | 812 个任务 | 让 Agent 像人一样浏览网页、点击按钮、填表单 |
| **SWE-bench** | 代码修复 | 2294 个真实 Bug | 给 Agent 一个 GitHub 上的真实 Bug，看能不能修好 |
| **ToolBench** | 工具使用 | 16000+ API | 考 Agent 会不会正确调用各种工具（搜索引擎、计算器等） |
| **API-Bank** | API 调用 | 73 个 API | 专门考 Agent 调用接口的能力 |

**这些基准测试怎么用？**

以 AgentBench 为例（学术界最常用的综合评估工具）：

1. 克隆仓库到本地：`git clone https://github.com/THUDM/AgentBench.git`
2. 安装依赖：`pip install -r requirements.txt`
3. 配置要测试的 Agent（比如指定用 GPT-4）
4. 运行测试：`python run.py --config configs/agents/gpt-4.yaml --tasks all`
5. 查看报告：会输出每个任务的得分和总分

**自定义基准测试**：除了用现成的，你也可以根据自己的业务需求构建专属测试集。核心思路是：

1. 收集你的真实用户请求（比如 100 条典型问题）
2. 为每条请求准备好"正确答案"
3. 让 Agent 逐条处理，对比输出和正确答案
4. 统计通过率和平均分

## 测试策略（怎么测 Agent）

测试 Agent 和测试普通软件不一样，因为 Agent 的行为不是完全确定的——同样的输入可能产生不同的输出。所以我们需要分层测试。

### 1. 单元测试 —— 测零件

Agent 由很多组件组成（工具、记忆、提示词等），单元测试就是单独测试每个组件是否正常工作。

**就像造车时，先单独测试发动机、刹车、转向是否正常。**

具体测试内容举例：
- **工具测试**：搜索工具能否正常返回结果？计算器能否正确计算？
- **记忆测试**：存入一段对话后，能否正确回忆起来？
- **提示词测试**：给定输入，LLM 是否按照预期格式输出？

### 2. 集成测试 —— 测组装

把多个组件组合在一起，测试它们之间的协作是否正常。

**就像车组装好了，测试发动机+变速箱+轮子配合后，车能不能正常跑。**

重点验证：
- Agent 收到用户消息后，是否正确调用了工具？
- 工具返回结果后，Agent 是否正确解析并生成了最终回答？
- 多轮对话中，记忆是否正确传递？

### 3. 端到端测试 —— 模拟真实用户

完全模拟真实用户的使用场景，从头到尾走一遍完整流程。

**就像把车开上路，测试在真实路况下的综合表现。**

测试场景示例：
- 用户说"帮我订明天去北京的机票"，Agent 是否完成了搜索→比价→预订的完整流程？
- 用户中途改主意说"不要明天的了，后天的呢"，Agent 是否正确理解并调整？

### 4. 压力测试 —— 考极限

把 Agent 推到极限，看它在哪里崩溃。

**就像测试汽车在极端条件下的表现：高温、高速、颠簸路面。**

常见压力测试：
- **并发测试**：同时发 100 个请求，Agent 能全部正常处理吗？还是有的会超时？
- **长对话测试**：连续聊 50 轮，Agent 的回答质量会下降吗？记忆会不会溢出？
- **恶意输入测试**：发送超长文本、空消息、特殊字符，Agent 会不会崩溃？

## 持续评估 —— 不是一次性的考试

Agent 评估不是"上线前测一次就完事了"，而是需要持续进行。

### CI/CD 集成

CI/CD（持续集成/持续部署）是软件开发中的自动化流程。在 Agent 项目中，它的意思是：每次你提交代码修改后，自动运行测试，确保没有破坏已有功能。

**工作流程**：

```
开发者修改代码 → 提交到 Git → 自动触发测试流程
                                    ↓
                            运行单元测试
                                    ↓
                            运行集成测试
                                    ↓
                            运行基准评估
                                    ↓
                        全部通过 → 自动部署
                        有失败 → 通知开发者修复
```

### 回归测试

"回归"的意思是：每次修改后，确认之前能正常工作的功能仍然正常。

举个例子：你优化了 Agent 的搜索功能，但同时可能不小心破坏了记忆功能。回归测试就是为了提前发现这类"改了 A 却坏了 B"的问题。

**操作方式**：
- 运行全部测试（`pytest tests/ -v`）：完整检查一遍
- 只运行之前失败的测试（`pytest tests/ --lf`）：快速验证修复是否有效
- 运行特定测试（`pytest tests/test_agent.py::test_search`）：精准测试某个功能

## 实战案例：构建一个简单的 Agent 评估器

理解了以上概念后，来看一个实际的评估器是怎么工作的。

**核心思路**：准备一组测试用例，逐个让 Agent 执行，然后根据预设的评分规则打分，最后汇总成报告。

**评估流程拆解**：

```
步骤 1：准备测试用例
  用例 1：输入="1+1等于几"，期望答案="2"
  用例 2：输入="北京天气"，期望答案包含"天气"
  用例 3：输入="订机票"，期望调用工具="search_flights"
  ...

步骤 2：逐个执行
  对每个用例：
    - 把输入发给 Agent
    - 收集 Agent 的输出（包括最终回答、调用了哪些工具、是否成功）

步骤 3：打分
  每个用例满分 100 分，分配如下：
    - 任务成功完成（没有报错）→ 40 分
    - 输出包含期望内容 → 40 分
    - 工具调用次数不超过上限 → 20 分

步骤 4：生成报告
  - 总用例数：50
  - 平均分：76
  - 通过率（≥60 分）：85%
  - 详细结果：每个用例的得分和失败原因
```

## 最佳实践

1. **建立测试用例库**：从真实用户请求中收集测试用例，而不是凭空编造。真实的用户请求才能暴露真实的问题
2. **多维度评估**：不要只看准确率。一个 Agent 可能答案正确，但消耗了 10 倍的成本，或者响应要 30 秒
3. **自动化优先**：能写成自动测试的，就不要靠人手动测。人工评估留给最需要主观判断的场景
4. **定期评估**：大模型更新（比如 GPT-4 升级到 GPT-5）后，一定要重新跑一遍评估，因为模型行为可能变化
5. **记录历史**：每次评估的结果保存下来，形成趋势图。这样才能看出 Agent 是在进步还是退步
