# 记忆系统

记忆系统决定了 Agent 能不能持续保持上下文、复用历史经验、以及在多轮任务中维持稳定性。

很多 Agent 看起来"不聪明"，问题并不在模型，而在它没有记住该记住的信息，或者在错误的时机取回了错误的记忆。

## 为什么记忆对 Agent 至关重要

LLM 本身是无状态的——每次调用都是独立的，不会"记得"上一次对话发生了什么。Agent 要想表现得像一个有经验的助手，就需要在模型之外构建一套记忆机制，让它能够：

- 跨会话保持用户偏好和历史上下文
- 从过去的成功/失败中学习，避免重复犯错
- 在长期任务中维持目标和中间状态

这本质上是把"上下文管理"从模型内部的 token 窗口，扩展到模型外部的持久化存储。

## 🧠 记忆的类型

从认知科学借鉴的分类方式，Agent 的记忆通常分为两大类：

### 短期记忆（Short-term Memory）

短期记忆对应人类的"工作记忆"——你正在思考的、刚听到的、暂时需要保持活跃的信息。在 Agent 中，它就是当前会话的上下文窗口。

**为什么需要短期记忆**：
- LLM 的上下文窗口有限（4K~128K tokens 不等），不可能把所有历史都塞进去
- 当前对话的连贯性依赖于最近几轮的交互内容
- 短期记忆读取速度快，不需要额外的检索开销

**关键设计考量**：
- **窗口大小**：保留多少轮对话？太少会丢失上下文，太多会浪费 token
- **淘汰策略**：满了之后怎么处理？简单的 FIFO，还是基于重要性评分？
- **摘要压缩**：是否在淘汰前先用 LLM 生成摘要，保留核心信息？

**实现方式**：
```python
from langchain_community.memory import ConversationBufferMemory

# 简单的对话记忆
memory = ConversationBufferMemory(
    memory_key="chat_history",
    return_messages=True,
    max_token_limit=2000  # 限制 token 数量
)

# 保存对话
memory.save_context(
    {"input": "你好，我想了解 AI Agent"},
    {"output": "AI Agent 是..."}
)

# 获取历史
history = memory.load_memory_variables({})
```

### 长期记忆（Long-term Memory）

长期记忆对应人类的"语义记忆"和"情景记忆"——你学过的知识、过去的经历、掌握的技能。在 Agent 中，它通过向量数据库实现持久化存储。

**为什么需要长期记忆**：
- 用户的偏好、历史交互、项目上下文等信息需要跨会话保留
- Agent 在执行任务中积累的经验应该被复用
- 随着使用时间增长，记忆库会越来越有价值

**向量数据库的作用**：
- 将文本转换为高维向量（embedding），捕捉语义信息
- 通过余弦相似度等度量找到与查询最相关的记忆
- 支持百万级记忆的高效检索

**实现方式**：
```python
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings

# 初始化向量存储
embeddings = OpenAIEmbeddings()
vectorstore = Chroma(
    embedding_function=embeddings,
    persist_directory="./long_term_memory"
)

# 添加记忆
vectorstore.add_texts(
    texts=["用户偏好 Python 编程", "项目截止日期是下周五"],
    metadatas=[{"type": "preference"}, {"type": "deadline"}]
)

# 检索记忆
results = vectorstore.similarity_search(
    query="用户的技术偏好",
    k=3,
    filter={"type": "preference"}
)
```

## 一个更实用的理解

记忆系统真正要解决的不是"存多少"，而是三件事：

1. **存什么**：不是所有对话都值得记住。需要判断哪些是关键信息（用户偏好、决策依据、任务状态），哪些是噪音（闲聊、重复确认）
2. **什么时候取**：在什么时机触发记忆检索？每次对话都查，还是只在特定场景（如用户提到"之前"、"上次"时）才查？
3. **取出来后如何影响当前任务**：检索到的记忆是直接拼接到 prompt 里，还是经过筛选、排序、摘要后再注入？

这三个问题的答案，直接决定了 Agent 的记忆系统是"有用"还是"添乱"。

## 📦 记忆系统架构

```
┌─────────────────────────────────────────────────┐

│                                                  │

│  │   短期记忆层     │    │   长期记忆层     │     │
│  │  (上下文窗口)    │    │  (向量数据库)    │     │
│  │                 │    │                 │     │
│  │ - 当前对话       │    │ - 历史经验       │     │
│  │ - 最近操作       │    │ - 用户信息       │     │
│  │ - 临时变量       │    │ - 知识库         │     │
│  └────────┬────────┘    └────────┬────────┘     │
│           │                      │               │
│           │    ┌────────────┐    │               │
│           └───→│  记忆管理   │←───┘               │
│                │   器       │                     │
│                └─────┬──────┘                     │
│                      │                            │
│           ┌──────────┼──────────┐                 │
│           │          │          │                 │
│    ┌──────▼──────┐ ┌─▼────────┐ │                 │
│    │   编码      │ │ 检索     │ │                 │
│    │  Encoding   │ │ Retrieval│ │                 │
│    └─────────────┘ └──────────┘ │                 │
│                                  │                 │
└─────────────────────────────────────────────────┘
```

**架构解读**：

- **短期记忆层**：充当"缓冲区"，存放当前活跃上下文，读写速度快但容量受限
- **长期记忆层**：充当"档案库"，存放经过筛选的持久化记忆，容量大但有检索开销
- **记忆管理器**：核心调度组件，决定何时将短期记忆"固化"到长期存储，何时从长期存储"唤醒"相关记忆
- **编码（Encoding）**：将文本转换为向量表示，是检索的基础
- **检索（Retrieval）**：根据查询找到最相关的记忆，通常用 ANN（近似最近邻）算法

## 🔧 记忆操作

### 1. 写入记忆

写入记忆的核心问题是"何时固化"——什么时候把短期记忆转存到长期存储？

常见的固化触发条件：
- **容量触发**：短期记忆满了，淘汰最旧的条目
- **会话结束**：用户退出对话时，批量转存
- **重要性触发**：LLM 判断当前信息值得长期保留
- **手动触发**：用户明确说"记住这个"

```python
class MemoryManager:
    def __init__(self, vectorstore, short_term_limit=10):
        self.vectorstore = vectorstore
        self.short_term = []
        self.limit = short_term_limit
    
    def add(self, content: str, metadata: dict = None):
        """添加记忆"""
        # 添加到短期记忆
        self.short_term.append({
            "content": content,
            "metadata": metadata or {},
            "timestamp": time.time()
        })
        
        # 维护短期记忆大小
        if len(self.short_term) > self.limit:
            # 将最旧的记忆转移到长期记忆
            old_memory = self.short_term.pop(0)
            self.consolidate(old_memory)
    
    def consolidate(self, memory):
        """将记忆转移到长期存储"""
        self.vectorstore.add_texts(
            texts=[memory["content"]],
            metadatas=[memory["metadata"]]
        )
```

### 2. 检索记忆

检索是记忆系统中最关键也最难的环节。难点在于：

- **相关性判断**：语义相似 ≠ 实际相关。用户说"上次那个项目"，需要结合时间、上下文来判断指的是哪个
- **信息融合**：短期记忆和长期记忆可能包含矛盾信息，需要决定优先级
- **检索数量**：返回太多会撑爆上下文，返回太少会遗漏关键信息

常见的检索策略：
- **纯语义检索**：用 embedding 相似度排序
- **混合检索**：结合关键词匹配 + 语义相似度
- **时间衰减**：越近的记忆权重越高
- **类别过滤**：根据当前任务类型过滤记忆类别

```python
def retrieve_memories(query: str, k: int = 5):
    """检索相关记忆"""
    # 从短期记忆搜索
    short_term_results = []
    for mem in short_term:
        similarity = calculate_similarity(query, mem["content"])
        short_term_results.append((mem, similarity))
    
    # 从长期记忆搜索
    long_term_results = vectorstore.similarity_search(query, k=k)
    
    # 合并结果
    all_results = short_term_results + long_term_results
    all_results.sort(key=lambda x: x[1], reverse=True)
    
    return all_results[:k]
```

### 3. 记忆更新

记忆不是只增不减的。当信息发生变化时（用户偏好改变、项目状态更新），旧记忆需要被更新而非简单追加。

更新策略有两种：
- **原地更新**：删除旧记忆，写入新内容。简单直接，但丢失历史版本
- **版本管理**：保留旧版本，标记为过期。可以追溯变化过程，但增加存储开销

```python
def update_memory(memory_id: str, new_content: str):
    """更新记忆"""
    # 删除旧记忆
    vectorstore.delete(ids=[memory_id])
    
    # 添加新记忆
    vectorstore.add_texts(
        texts=[new_content],
        ids=[memory_id]
    )
```

### 4. 记忆遗忘

遗忘不是 bug，而是 feature。人脑之所以能高效运作，正是因为会主动遗忘不重要的信息。Agent 的记忆系统同样需要遗忘机制：

- **避免信息过载**：太多过时记忆会干扰检索准确性
- **保护隐私**：敏感信息应该有生命周期，不能永久保留
- **保持相关性**：用户五年前的偏好可能早已改变

遗忘策略：
- **时间衰减**：超过一定时间未被检索的记忆自动清理
- **访问频率**：长期未被访问的记忆优先淘汰
- **重要性评分**：LLM 评估的记忆重要性低的优先遗忘

```python
def forget_old_memories(days: int = 30):
    """遗忘旧记忆"""
    cutoff_time = time.time() - (days * 24 * 3600)
    
    old_memories = vectorstore.get(
        where={"timestamp": {"$lt": cutoff_time}}
    )
    
    if old_memories:
        vectorstore.delete(ids=old_memories["ids"])
```

## 💡 记忆优化策略

### 1. 记忆压缩

当记忆积累到一定量，直接塞进 prompt 会导致 token 浪费和注意力分散。压缩策略的核心思想是：用更少的 token 保留更多的有效信息。

压缩的几种方式：
- **摘要压缩**：用 LLM 将多条相关记忆合并为一段摘要
- **关键词提取**：只保留记忆的核心关键词，丢弃修饰性内容
- **结构化提取**：将非结构化记忆转为结构化数据（如 JSON）

```python
from typing import List
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4", temperature=0)

def compress_memory(memories: List[str], llm) -> str:
    """压缩多条记忆为摘要"""
    prompt = f"""
    将以下记忆压缩为简洁的摘要：
    
    {chr(10).join(memories)}
    
    摘要：
    """
    return llm.invoke(prompt).content
```

### 2. 记忆分类

不同类型的记忆有不同的生命周期和使用场景。分类可以让检索更精准、淘汰更合理。

分类的价值：
- **精准检索**：用户问技术问题时，只检索 SKILL 和 FACT 类记忆，不混入闲聊
- **差异化保留**：PREFERENCE 类记忆应该长期保留，CONTEXT 类记忆可以在会话结束后清理
- **冲突解决**：当新记忆和旧记忆矛盾时，根据类别决定优先级（如新的 PREFERENCE 优先于旧的）

```python
from enum import Enum
from typing import List
from langchain_openai import ChatOpenAI

class MemoryCategory(Enum):
    FACT = "fact"          # 事实信息
    PREFERENCE = "preference"  # 用户偏好
    EXPERIENCE = "experience"  # 经验教训
    SKILL = "skill"        # 技能知识
    CONTEXT = "context"    # 上下文信息

def categorize_memory(content: str, llm) -> MemoryCategory:
    """自动分类记忆"""
    prompt = f"""将以下记忆分类为：FACT/PREFERENCE/EXPERIENCE/SKILL/CONTEXT
    
内容：{content}

只返回类别名称："""
    category = llm.invoke(prompt).content.strip().upper()
    try:
        return MemoryCategory(category)
    except ValueError:
        return MemoryCategory.CONTEXT
```

### 3. 记忆关联

孤立的记忆碎片价值有限。当记忆之间建立关联后，Agent 可以进行"联想式检索"——从一个记忆出发，找到相关的其他记忆。

关联的类型：
- **因果关联**："因为用户偏好 X，所以之前选择了 Y"
- **时序关联"："在做了 A 之后，又做了 B"
- **相似关联**："这个任务和之前的 Z 任务很像"

```python
def link_memories(memory1: str, memory2: str, relation: str):
    """建立记忆之间的关联"""
    # 在元数据中记录关联
    metadata = {
        "linked_to": memory2,
        "relation": relation
    }
    vectorstore.update_metadata(memory1, metadata)
```

## 🎯 实践：对话记忆 Agent

下面是一个完整的对话 Agent 示例，展示了短期记忆如何在对话中保持上下文：

**设计要点**：
- 使用 `ConversationBufferWindowMemory` 只保留最近 5 轮对话，避免上下文膨胀
- 通过 `MessagesPlaceholder` 将历史对话注入 prompt，让 LLM 能看到之前的交互
- `memory_key="chat_history"` 是记忆在 prompt 中的变量名，需要与 prompt 模板匹配

```python
from langchain_openai import ChatOpenAI
from langchain_community.memory import ConversationBufferWindowMemory
from langchain.agents import AgentExecutor, create_tool_calling_agent
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder

# 创建带记忆的 Agent
llm = ChatOpenAI(model="gpt-4", temperature=0)
memory = ConversationBufferWindowMemory(
    k=5,  # 保留最近 5 轮对话
    return_messages=True,
    memory_key="chat_history"
)

# 创建提示词模板
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是一个有帮助的助手，记住用户的偏好和历史对话"),
    MessagesPlaceholder(variable_name="chat_history"),
    ("human", "{input}"),
    MessagesPlaceholder(variable_name="agent_scratchpad"),
])

# 创建 Agent
agent = create_tool_calling_agent(llm, [], prompt)
agent_executor = AgentExecutor(
    agent=agent,
    tools=[],
    memory=memory,
    verbose=True
)

# 对话会保持上下文
agent_executor.invoke({"input": "我喜欢吃川菜"})
agent_executor.invoke({"input": "推荐一家餐厅"})
# Agent 会记住用户喜欢川菜
```

## ⚠️ 注意事项

### 1. 隐私保护

记忆系统会积累大量用户数据，隐私保护是必须考虑的问题：

- **输入脱敏**：写入记忆前，自动识别并脱敏身份证号、手机号、邮箱等 PII（个人可识别信息）
- **访问控制**：不同用户的记忆应该完全隔离，不能互相检索
- **删除权**：用户应该能够清除自己的所有记忆（GDPR 合规）

```python
def sanitize_memory(content: str) -> str:
    """清理敏感信息"""
    # 移除个人信息
    content = re.sub(r'\b\d{18}\b', '[ID]', content)  # 身份证号
    content = re.sub(r'\b1[3-9]\d{9}\b', '[PHONE]', content)  # 手机号
    
    return content
```

### 2. 记忆一致性

当 Agent 从多个来源获取信息时，可能会产生矛盾的记忆。一致性校验的作用是：

- **检测冲突**：新记忆是否与已有记忆矛盾
- **解决策略**：时间更新优先？可信度高的优先？还是提示用户确认？
- **合并重复**：语义相同但表述不同的记忆应该合并，避免冗余

```python
def check_consistency(new_memory: str, existing_memories: List[str], llm) -> bool:
    """检查新记忆与现有记忆是否一致"""
    prompt = f"""检查以下记忆是否一致：
    
新记忆：{new_memory}
现有记忆：{chr(10).join(existing_memories)}

是否存在矛盾？只回答"是"或"否"："""
    result = llm.invoke(prompt).content.strip()
    return "否" in result
```

### 3. 存储效率

Embedding 模型的选择直接影响存储成本和检索质量：

- **模型维度**：高维向量（如 1536 维的 OpenAI ada-002）语义表达力强，但存储和计算开销大
- **开源替代**：`sentence-transformers/all-MiniLM-L6-v2`（384 维）在很多场景下性价比更高
- **量化压缩**：将 float32 向量量化为 int8，可以将存储缩小 4 倍，精度损失通常可接受

```python
# 使用更高效的嵌入模型
from langchain_community.embeddings import HuggingFaceEmbeddings

embeddings = HuggingFaceEmbeddings(
    model_name="sentence-transformers/all-MiniLM-L6-v2"
)
```

## 📊 记忆系统评估

| 指标 | 描述 | 目标值 |
|------|------|--------|
| 检索准确率 | 检索到的记忆相关性 | >85% |
| 检索延迟 | 记忆检索时间 | <100ms |
| 存储效率 | 每条记忆的平均大小 | <1KB |
| 记忆容量 | 可存储的记忆数量 | >100 万 |

**评估方法说明**：
- **检索准确率**：通过人工标注或 LLM 评估，判断检索结果与查询的相关性
- **检索延迟**：从查询发出到结果返回的时间，影响用户体验
- **存储效率**：包括向量本身 + 元数据 + 原始文本的总大小
- **实际调优建议**：不要只追求单一指标。检索准确率和延迟往往需要权衡——提高准确率可能需要更大模型或更多检索数量，会增加延迟

## 🚀 进阶：分层记忆

受认知科学中"多重记忆系统"理论启发，可以将记忆细分为三个层次：

- **情景记忆（Episodic）**：记录具体事件，如"2024年3月15日用户询问了关于 RAG 的问题"。特点是有时空标记，可以回答"什么时候做过什么"
- **语义记忆（Semantic）**：抽象化的知识，如"用户是后端工程师，偏好 Go 语言"。特点是去时空化，是多条情景记忆的归纳
- **程序记忆（Procedural）**：操作性知识，如"处理用户投诉时应该先道歉再调查"。特点是隐式知识，不需要显式检索就能影响行为

分层的好处是不同层次有不同的更新频率和检索策略，可以更精细地管理记忆生命周期。

```python
class HierarchicalMemory:
    def __init__(self):
        self.episodic = VectorStore()  # 事件记忆
        self.semantic = VectorStore()  # 语义记忆
        self.procedural = VectorStore()  # 程序记忆
    
    def store(self, content: str, memory_type: str):
        """存储记忆到对应层级"""
        if memory_type == "episodic":
            self.episodic.add_texts([content])
        elif memory_type == "semantic":
            self.semantic.add_texts([content])
        elif memory_type == "procedural":
            self.procedural.add_texts([content])
    
    def retrieve(self, query: str, memory_types: List[str] = None):
        """从指定层级检索记忆"""
        results = []
        
        for mem_type in (memory_types or ["episodic", "semantic", "procedural"]):
            store = getattr(self, mem_type)
            results.extend(store.similarity_search(query, k=3))
        
        return results
```
