# Multi-Agent 协作

> 多个 Agent 协同工作，完成单个 Agent 无法完成的复杂任务。

---

## 为什么需要 Multi-Agent

单个 Agent 的局限性：

- **上下文窗口有限**：复杂任务会超出单次对话容量
- **专业能力有限**：一个 Agent 难以精通所有领域
- **并行能力有限**：串行处理效率低
- **视角单一**：缺乏多角度思考

Multi-Agent 通过分工协作解决这些问题。

## 核心架构模式

### 1. 主从模式（Hierarchical）

```
┌─────────────┐
│  主控 Agent  │
└──────┬──────┘
       │
   ┌───┴───┬───────┐
   ▼       ▼       ▼
┌─────┐ ┌─────┐ ┌─────┐
│Agent│ │Agent│ │Agent│
│  A  │ │  B  │ │  C  │
└─────┘ └─────┘ └─────┘
```

**特点**：主控 Agent 负责任务分配和结果汇总，子 Agent 负责执行具体任务。

**适用场景**：任务可明确拆分，需要统一协调。

### 2. 对等模式（Peer-to-Peer）

```
┌─────┐     ┌─────┐
│Agent│ ◄─► │Agent│
│  A  │     │  B  │
└──┬──┘     └──┬──┘
   │           │
   └─────┬─────┘
         ▼
      ┌─────┐
      │Agent│
      │  C  │
      └─────┘
```

**特点**：Agent 之间平等交互，自由协商。

**适用场景**：需要多轮讨论、辩论、评审。

### 3. 流水线模式（Pipeline）

```
┌─────┐    ┌─────┐    ┌─────┐    ┌─────┐
│Agent│ ► │Agent│ ► │Agent│ ► │Agent│
│  1  │    │  2  │    │  3  │    │  4  │
└─────┘    └─────┘    └─────┘    └─────┘
  输入      处理       验证       输出
```

**特点**：每个 Agent 处理一个阶段，依次传递。

**适用场景**：任务有明确的阶段划分。

### 4. 群体智能模式（Swarm）

```
     ┌─────┐
     │Agent│
     └──┬──┘
        │
   ┌────┼────┐
   ▼    ▼    ▼
┌─────┐┌─────┐┌─────┐
│Agent││Agent││Agent│
└─────┘└─────┘└─────┘
   ▲    ▲    ▲
   └────┼────┘
        │
     ┌──┴──┐
     │Agent│
     └─────┘
```

**特点**：大量简单 Agent 协同，涌现出复杂行为。

**适用场景**：模拟、搜索、优化问题。

## 通信机制

### 1. 直接消息传递

```python
# Agent A 向 Agent B 发送消息
message = {
    "from": "agent_a",
    "to": "agent_b",
    "type": "request",
    "content": "请分析这份数据"
}
agent_b.receive(message)
```

### 2. 共享黑板（Blackboard）

```python
# 所有 Agent 读写同一个共享空间
blackboard = {
    "task_status": {},
    "shared_data": {},
    "results": {}
}

# Agent A 写入
blackboard["results"]["analysis"] = result

# Agent B 读取
data = blackboard["results"]["analysis"]
```

### 3. 发布-订阅（Pub-Sub）

```python
# Agent 订阅感兴趣的事件
agent.subscribe("data_ready", handle_data)

# 其他 Agent 发布事件
agent.publish("data_ready", {"data": data})
```

## 常见协作场景

### 场景 1：研究讨论

```
用户问题 → 主持人 Agent
              │
    ┌─────────┼─────────┐
    ▼         ▼         ▼
 专家A     专家B     专家C
 (支持)   (反对)   (中立)
    │         │         │
    └─────────┼─────────┘
              ▼
         总结 Agent → 最终答案
```

### 场景 2：代码开发

```
需求 → 产品经理 Agent
           │
    ┌──────┼──────┐
    ▼      ▼      ▼
  架构师  开发者  测试者
    │      │      │
    └──────┼──────┘
           ▼
      代码审查 Agent → 最终代码
```

### 场景 3：内容创作

```
主题 → 策划 Agent
           │
    ┌──────┼──────┐
    ▼      ▼      ▼
  调研员  写手   编辑
    │      │      │
    └──────┼──────┘
           ▼
      审核 Agent → 最终文章
```

## 关键挑战

### 1. 一致性问题

不同 Agent 可能对同一问题有不同理解。

**解决方案**：
- 定义清晰的角色和职责
- 建立共享的术语表
- 使用主控 Agent 统一协调

### 2. 死锁问题

Agent 之间互相等待，导致任务停滞。

**解决方案**：
- 设置超时机制
- 引入仲裁 Agent
- 使用异步通信

### 3. 成本控制

多 Agent 意味着多次 LLM 调用，成本增加。

**解决方案**：
- 按需激活 Agent
- 使用缓存减少重复调用
- 选择合适的模型（小任务用小模型）

---

## 最佳实践

1. **从简单开始**：先用 2-3 个 Agent 验证可行性
2. **明确分工**：每个 Agent 职责单一、边界清晰
3. **定义接口**：消息格式、协议标准化
4. **监控协作**：记录 Agent 之间的交互过程
5. **容错设计**：Agent 失败时有降级方案
